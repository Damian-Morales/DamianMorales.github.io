Elastic Cloud SIEM Lab: PowerShell Activity Detection

Overview

In this mini-SIEM build, I deployed an Elastic Cloud deployment, enrolled a Windows host with Elastic Agent, ingested Windows Security/PowerShell logs, turned on prebuilt detection rules, and then generated benign PowerShell activity to validate end-to-end detection and investigation in Kibana. The test successfully produced alerts, which I explored using the Investigate in timeline workflow.

Objectives

  • Spin up Elastic Cloud & configure the Windows integration.
  • Enroll a Windows endpoint and confirm healthy telemetry.
  • Enable Elastic prebuilt rules related to authentication and PowerShell.
  • Generate test PowerShell events and confirm alerting.
  • Investigate alerts via host insights and Timeline.

Environment

  • Elastic Cloud (Kibana + Elasticsearch)
  • Elastic Agent on Windows with system and Windows integrations
  • Data streams: Windows Security, PowerShell/Script Block, Sysmon (where available)

Build & Configure

Detections & Investigation

I generated benign PowerShell activity (e.g., powershell -EncodedCommand ...) to create Microsoft-Windows-PowerShell provider events. Kibana raised alerts from my test rule “PowerShell Activity Detected” and related prebuilt logic.

Key Findings

  • Windows Security + PowerShell events successfully ingested and normalized.
  • Prebuilt rules and a custom test rule detected activity as expected.
  • Host insights and Timeline made it easy to pivot from alert → process → user.

What I Learned

  • End-to-end SIEM pipeline setup in Elastic Cloud and Fleet.
  • How to tune/target rules using event.provider, event.action, and message fields.
  • Practical investigation flows in Kibana (flyout → Take action → Timeline).

Next Steps

  • Expand to Sysmon and Filebeat for richer process/network telemetry.
  • Add case management automation and ticketing integrations.
  • Create dedicated rules for suspicious EncodedCommand usage and LOLBins.
