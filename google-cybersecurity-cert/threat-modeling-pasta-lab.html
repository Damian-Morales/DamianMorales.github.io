Threat Modeling with PASTA – My Work

Overview

In this activity, I performed a threat model of a new sneaker-buying app using the PASTA framework. I worked through all seven stages to identify business objectives, technical scope, threats, vulnerabilities, and controls. The result is a set of security requirements I can share with the development team.

Scenario

I’m part of the security team at a company building a mobile app for sneaker enthusiasts to buy and sell shoes. The app supports user sign-up, messaging between buyers and sellers, rating functionality, and payment processing. My goal was to use PASTA to uncover risks and recommend safeguards before launch.

Work Done

PASTA Stage My Response
I. Business Objectives I noted that the app must process secure transactions, enable real-time messaging, and comply with PCI-DSS regulations. I also highlighted the need for high availability and data privacy to maintain user trust.
II. Technical Scope I reviewed that the app uses a REST API for frontend-backend communication, PKI/TLS for encryption, SHA-256 for data integrity, and SQL database storage. I prioritized securing the API first—because it exposes the most functionality—and enforcing TLS to protect data in transit.
III. Decompose Application I examined the data-flow diagram showing “Search → Query DB → Return Listings.” I mapped out where user input enters the API, how credentials are validated, and where transaction records are stored.
IV. Threat Analysis I identified two key threats: external attackers launching SQL injection via the search endpoint, and insider misuse of administrative APIs to alter listings.
V. Vulnerability Analysis I found potential SQL injection in unsanitized queries and weak session tokens prone to hijacking. These flaws could let an attacker exfiltrate data or impersonate users.
VI. Attack Modeling I built an attack tree with root “Compromise App,” branches for “Inject SQL,” “Hijack Session,” and “Exploit API Keys,” and leaves detailing input vectors and session cookie theft.
VII. Controls & Mitigations I recommended using prepared statements/ORM to prevent injection, enforcing TLS with PKI, implementing MFA for administrative access, and performing regular code reviews and automated security testing.

Conclusion

By completing PASTA end-to-end, I captured the app’s security goals, mapped its architecture, listed threats and weaknesses, and proposed a layered defense strategy. These controls will help the development team build a more secure sneaker-marketplace app before it goes live.
